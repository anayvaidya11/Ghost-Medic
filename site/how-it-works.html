How it works

Four parts, one line of text between them.

Sensing on the wrist, thinking in the pack, one line of plain text between them ten times a second. Every stand-in is flagged.

01 · The wrist unit

Three sensors, and one thing it refuses to report

Three sensors, each answering a different question, driven by bare-metal C.

The partWhat it measuresWhat that is good for
BMP280Air pressure and temperatureAir thins predictably as you climb, so pressure gives you altitude. The temperature is the air around the device, not your body.
LIS3DHAcceleration, three directionsMotion, and the particular signature of a fall.
MAX30102Light reflected off skinRaw light readings, and deliberately nothing more. See below.

Height from air pressure

Run the pressure reading backwards to get height:

float bmp280_pressure_to_altitude(float pressure_pa) {
    return 44330.0f *
        (1.0f - powf(pressure_pa / BMP280_SEA_LEVEL_PA, 0.1903f));
}

The catch, in the code too: the sea-level reference is fixed but real pressure moves with the weather, so the figure is good for change ("climbed 300 m"), not absolute elevation. Not GPS.

What a fall looks like to a machine

Free fall reads near zero, landing spikes. Either alone is ordinary, so the firmware requires both, in that order, within 400 ms.

Label

A demonstration, not a medical-grade detector. The thresholds are untuned textbook values, and the code says so.

No heart rate

The optical sensor measures light off skin. That contains a pulse, but extracting it needs signal processing this project does not do, so the readings stay raw. Enforced in four places:

  • the firmware reports counts only;
  • the app has no heart-rate field;
  • the text sent to the model forbids inferring one;
  • a test fails if one ever appears.

An invented vital sign is worse than none.

02 · The link

A cable doing a radio's job

A short-range radio would do this; until it exists, a small program passes each line over a wire unchanged, so nothing downstream can tell a board from the simulator or a recording. It drops only the power-on greeting, which has no timestamp and so is not a reading.

03 · The pack computer

The model is told what the sensors saw, and what not to conclude

A small open model runs locally; the engineering is what it is told and forbidden. Every question carries the readings in words, each with its limits:

DERIVED READINGS:
- Altitude: 2.1 m (pressure-derived vs a fixed 101325 Pa sea-level
  reference; reliable for RELATIVE change only — NOT GPS or true elevation)
- Temperature: 21.0 °C (AMBIENT air at the device — NOT body temperature)
- Fall detected: YES (accelerometer free-fall→impact heuristic,
  not medical-grade)

RAW SIGNALS (non-diagnostic):
- Optical counts: red 10249, ir 10398. These are raw photodiode counts
  from the optical sensor. No pulse-extraction algorithm runs on this
  device: heart rate and blood-oxygen saturation (SpO2) are NOT
  available and must not be inferred from these numbers.

The last paragraph does the most: the light readings are not vital signs, and a failed sensor reads "unavailable", never zero.

Real output of services/sensorContext.ts, wrapped, markers removed.

One rule sits above the rest: if the readings disagree with the person, the person wins.

When the wristband speaks first

On a fall, the assistant asks on its own. One fall arrives as many identical readings, so the trigger fires only on the change to yes, then waits thirty seconds. One fall, one question.

const risingEdge = fallDetected && !prevFall;
prevFall = fallDetected;

if (!risingEdge) return false;
if (suppressed) return false; // consumed, not queued
if (nowMs - lastFiredMs < opts.cooldownMs) return false;

The real trigger code. The Proof chart shows what it saves: 2 model calls instead of 150 over a minute of replay.
